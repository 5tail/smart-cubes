# SPEC — 品牌 LOGO 展示（feature spec，決策層 2026-07-18）

展示頁（demo）連上方塊時，依品牌顯示各家 LOGO。本檔是此功能的規格與作圖規範，
供你備妥圖檔、供實作階段照做。**主 SPEC（統一介面/ADR）不受本檔影響**。

---

## 0. 一句話結論

LOGO 是**純展示層（demo）**的事，**不動凍結合約**。你只需要準備 4 個圖檔、放進
指定資料夾、用指定檔名；顯示邏輯由實作階段在 demo 端加。

---

## 1. 品牌分類與檔名（關鍵）

你要的四類裡，**奇藝(QY系) 與 魔方格(XMD系) 在程式碼是同一個 QiYi driver**
（`brand === 'qiyi'`），只靠裝置名稱前綴區分。因此本功能定義一組**「顯示品牌」key**
（只用於展示頁，與協議 `brand` 分開），對應如下：

| 顯示品牌 | key（檔名用） | 協議 brand | 判定依據（裝置名稱前綴） | 代表型號 |
|---|---|---|---|---|
| 淦源 GAN | `gan` | `gan` | `GAN` / `MG` / `AiCube` | GAN12 UI / i Carry |
| 魔域 MoYu | `moyu` | `moyu` | `WCU_MY3` | WeiLong AI V10/V11 |
| 奇藝 QiYi | `qiyi` | `qiyi` | `QY-QYSC` | QY-QYSC AI |
| 魔方格 MFG | `mofangge` | `qiyi` | `XMD`（如 `XMD-TornadoV4`） | Tornado V4 |

> 技術事實（不影響你的分類）：奇藝與魔方格共用同一組 QiYi 藍牙協議與金鑰，
> 在套件層是同一品牌；此處拆成兩個 LOGO 純粹是**行銷/顯示**區分，尊重你的市場認知。

### 檔名規範

```
gan.svg
moyu.svg
qiyi.svg
mofangge.svg
```

- **全小寫、ASCII、與上表 key 完全一致**（顯示邏輯會以 key 直接組路徑/查表）。
- 副檔名 `.svg`（首選）或 `.png`（見第 3 節）。若同時提供，實作以 `.svg` 優先。
- 未來新增品牌：新增一個 key + 一個檔案即可，命名比照。

## 2. 檔案放哪裡

```
demo/
  src/
    assets/
      brands/          ← 新增此資料夾，圖檔放這裡
        gan.svg
        moyu.svg
        qiyi.svg
        mofangge.svg
```

**為什麼放 `demo/src/assets/` 而非 `demo/public/`**：

- 本專案 demo 會部署到**子路徑**（GitHub Pages 的 `/smart-cubes/`、你的
  `/app/smartcubes/`）。放 `src/assets/` 並用 `import` 引入時，**Vite 會自動改寫成
  帶正確 base 的雜湊網址**，子路徑部署不會壞、且檔名帶 hash 可長快取。
- 放 `demo/public/` 則需要手動用 `import.meta.env.BASE_URL` 組路徑，漏了就在子路徑
  部署下 404 —— 這正是 DEPLOY.md 裡 base 那類坑，能避則避。

**不放套件層（`src/`）**：LOGO 只給展示頁看，套件（npm 發佈的 `src/`）不該夾帶
第三方商標圖檔（見第 5 節授權）。

## 3. 作圖規範

### 3.1 格式

- **首選 SVG**：向量、檔案小、任何解析度清晰。請確保有正確 `viewBox`、可縮放，
  文字轉外框（`<path>`，避免字型缺失）。
- **PNG 備案**（只有點陣素材時）：提供 **@2x** 尺寸（見下），透明背景，避免鋸齒。

### 3.2 尺寸 / 版面

展示頁連線行的 LOGO 目標**顯示高度約 24px**，未來品牌選擇卡片約 48px。請以「等高並排
不打架」為準則出圖：

- **統一視覺高度**：以 SVG `viewBox` 高度為基準，四家 LOGO 在**相同高度**下視覺重量相當
  （純圖示商標可做成方形 `viewBox="0 0 64 64"`；橫式字標可做 `viewBox="0 0 W 32"`）。
- **內距（safe area）**：圖形四周留約 8%「呼吸空間」，貼齊邊緣會顯得擁擠。
- PNG 版：短邊至少 **96px**（@2x 給 48px 卡片用），檔名同 key（`gan.png`…）。

### 3.3 顏色（最重要：demo 是深色底 `#0f1115`）

- LOGO 必須在**深色背景清楚可見**。純黑/深色墨的字標在深底會消失 —— 這類請改用
  **淺色/白色版**或帶自身底色的版本。
- 若你的素材只有深色版，兩條路擇一（實作會配合，但**請先在圖檔決定**）：
  1. 出一份**深底適用**的單檔（推薦，最省事）；或
  2. 出 `xxx-light.svg` / `xxx-dark.svg` 兩份，實作依主題切換
     （demo 目前恆為深色，暫時只會用到淺色那份，但雙份可面向未來淺色主題）。
- 若 LOGO 本身多色且對比足夠（在深底夠亮），單檔即可。

### 3.4 檔案大小預算

- SVG 每檔理想 **< 10 KB**（複雜漸層/內嵌點陣會爆量，請先 SVGO 壓縮）。
- PNG 每檔 **< 30 KB**。

### 3.5 提供前自檢清單

- [ ] 檔名 = 上表 key，全小寫 ASCII
- [ ] SVG 有 viewBox、文字已轉外框、跑過 SVGO
- [ ] 放到深色底 `#0f1115` 上清楚可見
- [ ] 四家並排等高、視覺重量相當
- [ ] 檔案大小在預算內

## 4. 顯示行為（實作階段做，供你了解範圍）

- demo 端新增純函式 `displayBrand(brand, deviceName)`：`qiyi` 且名稱前綴 `XMD` →
  `'mofangge'`，前綴 `QY-QYSC` → `'qiyi'`，其餘照 `brand`。**有 fixture 單元測試**
  （前綴映射鎖住行為）。
- 連上後，連線行 `#device-name` 前面插入對應 `<img>`（`alt` 為品牌中文名，無障礙）。
- 找不到對應圖檔時**優雅退場**：不顯示圖、只顯示文字（現狀），不報錯。
- 凍結合約 `src/core/types.ts` **零改動**；不新增 `brand` 列舉值。

## 5. ⚠️ 授權與商標（決策層須先拍板）

**這是本功能唯一需要你判斷的實質風險，請先確認再備圖。**

- 品牌 LOGO 是 GAN / 魔域 / 奇藝 / 魔方格各自的**註冊商標與著作**，
  **不屬於本專案 GPL-3.0 授權範圍**。把它們 commit 進公開 repo 等於再散佈他人商標。
- 用 LOGO **標示「本工具相容此品牌方塊」**，多數司法管轄屬「指名合理使用」
  （nominative fair use），一般可接受；但仍建議：
  1. **來源優先用官方品牌素材／新聞資料包**，避免來路不明或他人重繪版；
  2. LOGO **只進 demo、不進 npm 套件**（本 spec 的放置位置已確保這點）；
  3. 在 **`NOTICE.md` 記錄每個 LOGO 的來源與商標歸屬**（比照 csTimer 移植的登錄慣例）；
  4. 若對某家素材授權沒把握，該家先用**純文字品牌名**佔位，不放圖。
- 決策點：**你要親自確認手上這些圖檔可被公開散佈**。實作階段不會替你判斷授權，
  只會照你放進資料夾的檔案接線。

## 6. 「發佈最新版」的範圍澄清

LOGO 是 demo 的事，**demo 不在 npm 套件內**。因此本功能的「發佈」＝

- **重新部署展示頁**（GitHub Pages push main 自動部署；你的站照 DEPLOY.md 重 build 上傳）。
- **不需要** bump npm 套件版本（`package.json` 的 0.1.0 與 LOGO 無關）。

若你另有其他要進套件的改動想一起發 npm，那是獨立的一次 `npm publish`（見 BACKLOG
發佈項），與本功能分開處理。

## 7. 交付與驗收（實作階段）

1. 你備妥 4 圖檔放 `demo/src/assets/brands/`，確認第 5 節授權。
2. 實作：`displayBrand` 純函式 + fixture 測試；連線行插入 `<img>`；找不到圖優雅退場。
3. `NOTICE.md` 補 LOGO 來源與商標歸屬。
4. 綠燈（typecheck + test + demo build）→ commit → 更新 CHANGELOG → PR。
5. 實機/瀏覽器驗收：四類方塊（或改名模擬前綴）連上後各自 LOGO 正確、深底可見、
   找不到圖不報錯。
